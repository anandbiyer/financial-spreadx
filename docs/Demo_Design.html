Requirement A — PDF Page Classification (Digital / Scanned / Hybrid)

Problem: A single annual report can mix native-digital pages (text extractable by pdf-parse) with scanned inserts such as auditor signatures, stamp pages, or legacy comparative statements. Page-level classification is needed to avoid sending image pages to the text extraction pipeline and to flag which pages need OCR.

Solution design

Every page is classified before any AI call. Classification is fully deterministic — no Claude call needed. The rule set:

Implementation — lib/pdf/page-classifier.ts

// lib/pdf/page-classifier.ts
import pdfParse from 'pdf-parse';

export type PageClassification = 'digital' | 'scanned' | 'hybrid';

export interface ClassifiedPage {
  pageNumber: number;
  classification: PageClassification;
  wordCount: number;
  asciiRatio: number;
  textContent: string;
  requiresOCR: boolean;
}

export async function classifyPdfPages(pdfBuffer: Buffer): Promise<ClassifiedPage[]> {
  const pages: ClassifiedPage[] = [];

  // pdf-parse renders page by page when a pagerender callback is provided
  let currentPage = 0;
  await pdfParse(pdfBuffer, {
    pagerender: async (pageData: any) => {
      currentPage++;
      const text = (await pageData.getTextContent())
        .items.map((i: any) => i.str).join(' ');

      const words = text.trim().split(/\s+/).filter(w => w.length > 1);
      const wordCount = words.length;
      const asciiCount = [...text].filter(ch => ch.charCodeAt(0) < 128).length;
      const asciiRatio = text.length > 0 ? asciiCount / text.length : 0;

      let classification: PageClassification;
      if (wordCount >= 80 && asciiRatio >= 0.90) {
        classification = 'digital';
      } else if (wordCount >= 20) {
        classification = 'hybrid';
      } else {
        classification = 'scanned';
      }

      pages.push({
        pageNumber: currentPage,
        classification,
        wordCount,
        asciiRatio,
        textContent: classification !== 'scanned' ? text : '',
        requiresOCR: classification === 'scanned',
      });
    },
  });

  return pages;
}

Requirement B — Financial Page Filtering

Problem: Annual reports run 200–400 pages. Sending all pages to Claude for extraction is expensive and introduces noise from non-financial sections (risk disclosures, governance reports, audit committee reports). Only financial statement pages and their associated notes should be extracted.

Solution design

Two-stage filtering applied to all digital and hybrid pages after classification:

  1. Stage 1 — Heading keyword scan (deterministic, fast): Each page’s text is scanned for section heading patterns within the first 300 characters. Pages matching a financial section pattern are promoted to candidates.
  2. Stage 2 — Boundary detection: Candidates are grouped into contiguous sections (a balance sheet section typically runs 2–6 pages). A trailing window of up to 5 pages after a section heading is included to capture continuation pages.
  3. Note page detection: Pages beginning with “Note [N]” or “[N]. [Title]” in the heading area are tagged with their note number for entity linking.

Implementation — lib/pdf/page-filter.ts

export interface FilterResult {
  selectedPages: Map<PageSection, number[]>;
  notePageMap: Map<number, number[]>;   // note_number -> page_numbers
  filteredPageCount: number;
  totalPageCount: number;
  reductionRatio: number;               // e.g. 0.12 = 12% of pages retained
}

export function filterFinancialPages(
  classifiedPages: ClassifiedPage[]
): FilterResult {
  const digitalPages = classifiedPages.filter(p => p.classification !== 'scanned');
  const selected = new Map<PageSection, number[]>();
  const notePageMap = new Map<number, number[]>();

  for (const page of digitalPages) {
    const heading = page.textContent.slice(0, 400);
    for (const [section, patterns] of Object.entries(SECTION_PATTERNS)) {
      if (patterns.some(p => p.test(heading))) {
        const arr = selected.get(section as PageSection) ?? [];
        arr.push(page.pageNumber);
        selected.set(section as PageSection, arr);
        break;
      }
    }
    // Detect note pages
    const noteMatch = heading.match(/^note\s+(\d+)/i) ??
                      heading.match(/^(\d+)\.\s+[A-Z]/m);
    if (noteMatch) {
      const num = parseInt(noteMatch[1]);
      const pages = notePageMap.get(num) ?? [];
      pages.push(page.pageNumber);
      notePageMap.set(num, pages);
    }
  }

  // Expand each section by trailing-window (continuation pages)
  for (const [section, pages] of selected) {
    const expanded = expandWithContinuationPages(pages, digitalPages, 5);
    selected.set(section, expanded);
  }

  const allSelected = new Set([...selected.values()].flat());
  return {
    selectedPages: selected,
    notePageMap,
    filteredPageCount: allSelected.size,
    totalPageCount: classifiedPages.length,
    reductionRatio: allSelected.size / classifiedPages.length,
  };
}

Problem: Annual reports commonly contain both standalone (parent entity alone) and consolidated (parent + subsidiaries) financial statements — particularly common in Indian NBFC reports (T3) and UK holding companies (T5). Solution B — Multi-year column detection

The extraction prompt is enhanced to return all year columns detected, not just the two most recent. The column classifier then labels each column.

// lib/pdf/column-classifier.ts
export type ColumnType = 'actual' | 'budget' | 'forecast' | 'restated' | 'unknown';

export interface ColumnMetadata {
  label: string;        // raw column header, e.g. "2023" or "Year ended Mar 31, 2022"
  year: number;         // parsed year integer
  type: ColumnType;
}

const COLUMN_TYPE_PATTERNS: Record<ColumnType, RegExp[]> = {
  restated: [/(restated|re-stated|revised|as\s+restated)/i],
  budget:   [/(budget|budgeted)/i],
  forecast: [/(forecast|projected|projected|estimate)/i],
  actual:   [],  // default if none of the above match
};

export function classifyColumnHeaders(rawHeaders: string[]): ColumnMetadata[] {
  return rawHeaders.map(header => {
    const yearMatch = header.match(/20\d{2}|19\d{2}/);
    const year = yearMatch ? parseInt(yearMatch[0]) : 0;
    let type: ColumnType = 'actual';
    for (const [t, patterns] of Object.entries(COLUMN_TYPE_PATTERNS)) {
      if (patterns.some(p => p.test(header))) {
        type = t as ColumnType;
        break;
      }
    }
    return { label: header, year, type };
  });
}

Solution C — Actuals vs Forecast/Restated handling

The mapping engine ignores forecast/budget columns for V01–V12 validation checks (validation only runs against actual columns). Restated columns are extracted and stored but not used as the primary value for validation. The review workbench shows a column type badge next to each year header.

Requirement D — Entity Linking (Notes to Accounts)

Problem: Financial line items reference notes throughout (e.g., “Property, Plant & Equipment (Note 12)”). Currently note references are extracted as a raw string but not linked to note content. Entity linking builds a bidirectional index so every line item links to its full note text, and every note links to the line items that reference it.

Solution design

  1. Page filter (Requirement B) has already identified note pages and built a notePageMap: Map<number, number[]>.
  2. A new pipeline stage — Note Extraction — runs after financial row extraction. It iterates each note in the notePageMap and calls Claude to extract the note title, summary text, and any sub-tables.
  3. The extracted rows already have a note_ref field (e.g., “Note 12”). The entity linker parses the note number from this field and writes a foreign key relationship to the note_entries table.
  4. The review workbench shows a clickable “Note 12 →” link on any row with a note reference. Clicking opens a side-panel showing the full note text and any sub-tables.

Entity linker — lib/mapping/entity-linker.ts

// lib/mapping/entity-linker.ts
export function parseNoteNumber(noteRef: string | null): number | null {
  if (!noteRef) return null;
  const match = noteRef.match(/\d+/);
  return match ? parseInt(match[0]) : null;
}

export async function linkNotesToRows(
  documentId: string,
  extractedRows: ExtractedRow[],
  noteEntries: NoteEntry[]
): Promise<void> {
  const noteMap = new Map(noteEntries.map(n => [n.note_number, n.id]));

  for (const row of extractedRows) {
    const noteNum = parseNoteNumber(row.note_ref);
    if (noteNum && noteMap.has(noteNum)) {
      // Write bidirectional link: update note_entries.linked_row_ids
      await db.execute(sql`
        UPDATE note_entries
        SET linked_row_ids = array_append(linked_row_ids, ${row.id}::uuid)
        WHERE document_id = ${documentId} AND note_number = ${noteNum}
      `);
    }
  }
}

Requirement E — OCR for Scanned Pages

Problem: The page classifier (Req A) already identifies pages as scanned when word count is below 20 and the text layer is absent or corrupt. Without OCR, those pages are silently skipped, which means any financial data that appears only on scanned pages is lost. OCR closes this gap by rasterising scanned pages and sending the resulting images to Claude Vision for direct financial row extraction.

Chosen approach — Claude Vision extraction

Rather than a generic OCR engine (Tesseract, AWS Textract) followed by a text extraction pass, scanned pages are rasterised to PNG and sent directly to Claude using its vision capability. Claude receives the image and the same structured extraction prompt, returning the same ExtractedRow[] schema. This eliminates a pipeline stage, keeps the architecture uniform, and is specifically strong on financial table layouts.

  pdfBuffer: Buffer,
  pageNumber: number,
  scale = 2.0
): Promise<Buffer> {
  const pdf = await pdfjs.getDocument({ data: new Uint8Array(pdfBuffer) }).promise;
  const page = await pdf.getPage(pageNumber);
  const viewport = page.getViewport({ scale });

  const canvas = createCanvas(Math.floor(viewport.width), Math.floor(viewport.height));
  const ctx = canvas.getContext('2d') as any;

  await page.render({ canvasContext: ctx, viewport }).promise;
  await pdf.destroy();

  // Return PNG buffer at ~150-300 KB per page at 2x scale
  return canvas.toBuffer('image/png');
}

export async function rasterizePages(
  pdfBuffer: Buffer,
  pageNumbers: number[],
  scale = 2.0
): Promise<Map<number, Buffer>> {
  const result = new Map<number, Buffer>();
  for (const pageNum of pageNumbers) {
    result.set(pageNum, await rasterizePage(pdfBuffer, pageNum, scale));
  }
  return result;
}

export async function extractStatementFromImage(
  imageBuffer: Buffer,
  statementType: string,
  templateType: string,
  pageNumber: number
): Promise<z.infer<typeof extractedRowSchema>['rows']> {
  const base64 = imageBuffer.toString('base64');

  const response = await client.messages.create({
    model: 'claude-sonnet-4-5',
    max_tokens: 4096,
    messages: [{
      role: 'user',
      content: [
        {
          type: 'image',
          source: { type: 'base64', media_type: 'image/png', data: base64 },
        },
        {
          type: 'text',
          text: `This is page ${pageNumber} of a financial statement (${statementType}, template: ${templateType}).
Extract all financial line items as structured rows.
Return JSON matching this schema exactly:
{
  "rows": [
    {
      "raw_label": "string (verbatim label from the image)",
      "raw_values": { "2024": number | null, "2023": number | null },
      "section_path": ["string"],
      "indentation_level": 0-4,
      "is_subtotal": boolean,
      "note_ref": "string | null"
    }
  ]
}
Rules: raw_label must be the exact text from the image. Negative values use negative numbers, not parentheses. Return only JSON.`,
        },
      ],
    }],
  });

  const text = response.content.find(b => b.type === 'text')?.text ?? '{"rows":[]}';
  const clean = text.replace(/```json|```/g, '').trim();
  const parsed = extractedRowSchema.parse(JSON.parse(clean));
  return parsed.rows;
}

Section 05

Mapping Engine Design

The mapping engine is a standalone module — not entangled with the extraction layer. It transforms raw row labels into canonical financial concepts. Six internal modules in sequence.

Module 1 — Label Normalizer

Pure string transformations. Runs before any dictionary lookup.

  • Strip leading Roman/Arabic numerals: "VII. Profit for the year" → "profit for the year"
  • Remove note references: "Interest income (Note 21)" → "interest income"
  • Expand common abbreviations: PBT → profit before tax, PAT → profit after tax, EPS → earnings per share
  • Normalize whitespace, lowercase, strip parenthetical qualifiers like (Restated) or (net)
  • Handle multi-line labels common in Asian templates (translated text)

Module 2 — Canonical Dictionary

A JSON lookup table of ~400 label → canonical field mappings, organized by template family. Each entry includes a base match (exact normalized string) and optional partial matches (for label variants).

    "canonical_field": "net_income",

    "templates": ["T3"],

  }
}

Module 3 — Context-Aware Disambiguator

For labels that appear in multiple statements with different meanings. Uses: statement_type, section_path, row_neighbors, and template_type as context signals.

  • "other income" in income statement → other_income
  • "other income" appearing below "total assets" → likely a balance sheet subtotal, flag for review
  • "interest income" in cash flow statement (investing section) → interest_received_investing
  • "interest income" in income statement → interest_income

Module 4 — Hierarchy & Rollup Engine

Builds the statement tree from indentation levels and subtotal detection. Validates that child rows sum to their parent subtotals. Infers missing intermediate totals where the arithmetic is unambiguous.

Assigns a composite confidence score (0–1) to each mapped row. Score is a weighted combination of: dictionary match quality, context signal alignment, formula validation pass/fail, and historical analyst agreement rate for this template × label pair.

Review threshold: Rows with composite confidence < 0.80 are auto-flagged for analyst review. Rows with confidence ≥ 0.95 and V01–V09 passing are auto-approved. Target: <15% of rows require manual review for T1/T3/T5 templates.

export async function GET(req: NextRequest, { params }: { params: { id: string } }) {
  const [doc, rows] = await Promise.all([
    getDocumentById(params.id),
    getMappedRowsByDocument(params.id, 'reviewed'),
  ]);
  const buffer = await generateXlsx(doc, rows);
  const blob = await put(`exports/${params.id}-reviewed.xlsx`, buffer, { access: 'private' });
  return NextResponse.json({ url: blob.url, expiresAt: new Date(Date.now() + 7*24*60*60*1000) });
}
