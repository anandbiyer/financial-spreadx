Section 01

Business Context & Objectives

Financial spreading is the process by which a bank transforms a borrower’s raw financial statements into a standardised, comparable format that can be used for credit evaluation, financial ratio generation, and risk scoring. The current Financial SpreadX demo extracts and normalises financial data from annual report PDFs. This requirements document defines the next layer: mapping those extracted line items into a predefined standardised Chart of Accounts (CoA) and spreading them into an output that a bank’s credit system can directly consume.

Why this matters for credit analysis

  • Comparability: Different companies use entirely different labels for the same economic item (“Turnover”, “Net Sales”, “Revenue from Operations” all mean the same thing). The CoA normalises this.
  • Ratio generation: Financial ratios (DSCR, Current Ratio, Debt/EBITDA) require line items to be in the correct CoA bucket. A wrong mapping produces a wrong ratio.
  • Auditability: Every mapping decision must carry a probability score and a reason. The bank’s credit officer must be able to challenge any mapping.
  • Consistency: The same line item must always map to the same CoA entry across different companies, analysts, and time periods.
Source authority. The target CoA is defined in BS_PL_Line_Items_V1.xlsx (two sheets: BS Definitions, P&L Definitions). The mapping logic and quality standards are defined in Prompt_for_CoA_Mapping_V1.docx. Both files are the single source of truth for this feature. Do not deviate from them.

The spreading and COA mapping feature sits as a new Stage 11 after the existing 10-stage extraction pipeline. Once rows have been extracted, mapped to canonical fields, and validated (Stages 1–10), they are now also mapped to the standardised CoA and spread into the bank’s output format.

Three sub-features

#Sub-featureWhat it does

Spreading Engine: Architecture & Pipeline Stage

Where Stage 11 runs

Stage 11 is appended to the existing synchronous pipeline in app/api/documents/route.ts after Stage 10 (validation). It receives the full set of mapped_rows for the document and produces a spread_result record with one coa_mapping entry per row.

// (runs after Stage 10 validation)

// Stage 11: COA Mapping & Spreading
await db.update(documents).set({ status: 'spreading' }).where(eq(documents.id, doc.id));

const mappedRows = await getMappedRowsByDocument(doc.id);
const coaEntries = await getAllCoaReference();  // full reference table

const spreadResult = await runCOAMappingStage({
  documentId: doc.id,
  mappedRows,
  coaEntries,
  templateType: classification.template_type,
  statementScope: classification.statement_scopes,
});

// spreadResult contains:
// - coaMappings[]: one per row, with coa_id, confidence, rationale, mapping_source
// - unmappedItems[]: rows below confidence threshold
// - balanceCheck: { totalAssets, totalLiabilitiesEquity, isBalanced, difference }
// - subtotalChecks[]: each subtotal verified

await db.insert(spreadResults).values(spreadResult.summary);
await db.insert(coaMappings).values(spreadResult.coaMappings);
if (spreadResult.unmappedItems.length > 0) {
  await db.insert(unmappedItems).values(spreadResult.unmappedItems);
  await db.update(documents).set({ status: 'has_unmapped', unmapped_count: spreadResult.unmappedItems.length });
} else {
  await db.update(documents).set({ status: 'spread_complete' });

COA Mapping Engine entry point — lib/spreading/coa-mapper.ts

// lib/spreading/coa-mapper.ts
export async function runCOAMappingStage(input: {
  documentId: string;
  mappedRows: MappedRow[];
  coaEntries: CoaReference[];
  templateType: string;
  statementScope: string[];
}): Promise<SpreadResult> {

  const coaMappings: CoaMappingRecord[] = [];
  const unmappedItems: UnmappedItem[] = [];

  for (const row of input.mappedRows) {
    // Step 1: Check learned mappings first (no Claude call)
    const learned = await findLearnedMapping(row.canonical_field, row.raw_label, input.templateType);
    if (learned) {
      coaMappings.push({
        document_id: input.documentId,
        mapped_row_id: row.id,
        coa_id: learned.coa_id,
        confidence: learned.learned_confidence,
        rationale: learned.rationale,
        mapping_source: 'learned',
        learned_mapping_id: learned.id,
        learned_from_doc: learned.source_document,
        learned_at: learned.created_at,
      });
      continue;
    }

    // Step 2: Claude-based mapping
    const result = await mapRowToCoaWithClaude(row, input.coaEntries, input.templateType);

    if (result.confidence >= 0.60) {
      coaMappings.push({
        ...result,
        document_id: input.documentId,
        mapped_row_id: row.id,
        mapping_source: 'claude',
      });
    } else {
      // Below threshold — flag as unmapped
      unmappedItems.push({
        document_id: input.documentId,
        mapped_row_id: row.id,
        raw_label: row.raw_label,
        canonical_field: row.canonical_field,
        statement_type: row.statement_type,
        claude_suggestions: result.candidates,  // top 3 suggestions
        status: 'pending',
      });
    }
  }

  // Run balance sheet check
  const balanceCheck = checkBalanceSheetIdentity(coaMappings, input.coaEntries);
  const subtotalChecks = verifySubtotals(coaMappings, input.coaEntries);

  return { coaMappings, unmappedItems, balanceCheck, subtotalChecks };
}

Section 05

COA Mapping Module: Process, Confidence & Validation

Mapping process (from Prompt_for_CoA_Mapping_V1.docx)

The mapping logic is defined in the Word document and must be implemented exactly as specified. The following steps are performed for every row:

  1. CoA comprehension: Load the target CoA and understand the definition and spreading guidance for every entry.
  2. Financial statement classification gating: A Balance Sheet row must map to a BS-xxx CoA entry only. A P&L row must map to a PL-xxx CoA entry only. Cross-statement mapping is a hard error.
  3. Economic substance analysis: Analyse the company line item based on its economic substance, accounting treatment, recognition basis, measurement basis, and IFRS/GAAP meaning. Do not rely solely on the label text.
  4. Candidate evaluation: Compare the row against all valid CoA candidates on four dimensions: semantic similarity, accounting similarity, reporting similarity, and regulatory similarity.
  5. Best match selection: Select the best match based on accounting meaning, reporting treatment, and financial statement classification alignment — not textual similarity alone.
  6. Ambiguity penalisation: Penalise confidence score for: multiple valid CoA targets, missing definitions, industry-specific terminology, parent-child account conflicts, and aggregated line items.
  7. Rationale generation: Write a detailed rationale string explaining exactly why the mapping is correct. If ambiguity exists, explain it clearly.
  8. Balance sheet identity check: After all rows are mapped, verify Total Assets = Total Liabilities + Total Equity. Report discrepancies with the contributing line items.
  9. Subtotal consistency check: Verify that grouped CoA subtotals (e.g., Total Current Assets, Gross Profit, Total Operating Expenses) are arithmetically consistent.

Claude prompt for COA mapping — lib/spreading/map-row-to-coa.ts

// lib/spreading/map-row-to-coa.ts
export async function mapRowToCoaWithClaude(
  row: MappedRow,
  coaEntries: CoaReference[],
  templateType: string
): Promise<CoaMappingResult> {

  const relevantCoa = coaEntries.filter(
    c => c.statement === statementTypeToCoaStatement(row.statement_type)
  );

  const { object } = await generateObject({
    model: anthropic('claude-sonnet-4-5'),
    schema: z.object({
      coa_id:      z.string().regex(/^(BS|PL)-\d{3}$/),
      confidence:  z.number().min(0).max(1),
      rationale:   z.string().min(50),
      ambiguities: z.array(z.string()),
      candidates:  z.array(z.object({
        coa_id:    z.string(),
        score:     z.number(),
        reason:    z.string(),
      })).max(3),
    }),
    system: `You are an expert Chartered Accountant, IFRS specialist, Banking Finance SME,
and Chart of Accounts mapping expert. You map financial statement line items into a
predefined standardised COA for a bank's credit evaluation system.
Accuracy, explainability, consistency, and auditability are critical.
Perform semantic accounting analysis — do not rely solely on keyword matching.
The definition of each COA entry takes precedence over its title.`,
    prompt: `Map the following financial line item to the standardised COA.

COMPANY LINE ITEM:
- Raw label: "${row.raw_label}"
- Canonical field: "${row.canonical_field}"
- Statement type: ${row.statement_type}
- Statement scope: ${row.statement_scope}
- Template type: ${templateType}
- Values: ${JSON.stringify(row.canonical_values)}

TARGET COA ENTRIES (${relevantCoa.length} candidates, same statement only):
${relevantCoa.map(c => \`\${c.coa_id}: \${c.line_item_name}
  Definition: \${c.definition}
  Spreading guidance: \${c.spreading_guidance}\`).join('\
\n')}

MAPPING RULES:
1. This is a ${row.statement_type} item — map to ${statementTypeToCoaStatement(row.statement_type)} CoA entries only.
2. Focus on economic substance, not label similarity.
3. Penalise confidence if multiple CoA entries are plausible.
4. If the item is aggregated (covers multiple CoA entries), map to the most appropriate parent.
5. Provide top-3 candidates with individual scores.

Return: best coa_id, confidence (0-1), detailed rationale, ambiguities list, top-3 candidates.`,
  });

  return object;
}

Balance sheet validation — lib/spreading/balance-checks.ts

// lib/spreading/balance-checks.ts
export function checkBalanceSheetIdentity(
  mappings: CoaMappingRecord[],
  coa: CoaReference[]
): BalanceCheckResult {
  const assetIds = coa.filter(c => ['Current Assets','Non-Current Assets'].includes(c.broad_category))
                      .map(c => c.coa_id);
  const liabIds  = coa.filter(c => ['Current Liabilities','Non-Current Liabilities'].includes(c.broad_category))
                      .map(c => c.coa_id);
  const equityIds = coa.filter(c => c.broad_category === 'Equity').map(c => c.coa_id);

  const totalAssets = sumMappingValues(mappings.filter(m => assetIds.includes(m.coa_id)));
  const totalLiabs  = sumMappingValues(mappings.filter(m => liabIds.includes(m.coa_id)));
  const totalEquity = sumMappingValues(mappings.filter(m => equityIds.includes(m.coa_id)));
  const diff = totalAssets - (totalLiabs + totalEquity);

  return {
    totalAssets,
    totalLiabilities: totalLiabs,
    totalEquity,
    totalLiabilitiesAndEquity: totalLiabs + totalEquity,
    difference: diff,
    isBalanced: Math.abs(diff) / totalAssets < 0.001,  // 0.1% tolerance
    imbalanceContributors: diff !== 0 ? identifyImbalanceContributors(mappings) : [],
  };
}
Section 06

Unmapped Item Resolution Workflow

What triggers the unmapped workflow

Any row where Claude’s best mapping confidence is below 0.60 is placed in the unmapped_items table with status pending. The document status is set to has_unmapped and the analyst is directed to the Unmapped Resolver screen before the spread output is finalised.

Unmapped Resolver screen — /spread/[id]/unmapped

This is a dedicated screen (not the existing Review Workbench) built specifically for resolving CoA mapping ambiguities. It shows only the unmapped items from the current document.

Screen layout

  • Left panel (40%): The unmapped item details — raw label, canonical field, statement type, values, and a short explanation of why Claude could not map it with confidence (“Ambiguity: 3 valid CoA targets identified. Industry-specific terminology (NBFC).”)
  • Right panel (60%): The ranked CoA suggestions from Claude, displayed as selectable cards. Each card shows: CoA ID and name, Claude’s match score for this suggestion, the CoA definition excerpt, and the rationale for this particular candidate match. The analyst clicks one card to select it and optionally edits the rationale before confirming.
  • Bottom bar: “N items remaining” counter, Skip (defer for later), Confirm mapping button.

Suggestion generation

The top-3 candidate CoA entries are generated by Claude during Stage 11 and stored in unmapped_items.claude_suggestions (JSONB). No additional Claude call is needed on the Unmapped Resolver screen — the suggestions are pre-computed. The screen renders them directly from the database.

After resolution

When the analyst confirms a mapping for an unmapped item:

  1. A new record is inserted into coa_mappings with mapping_source = 'manual' and the analyst-confirmed CoA ID.
  2. The unmapped_items record is updated to status = 'resolved'.
  3. A new record is inserted into learned_mappings (see Section 07).
  4. When all unmapped items are resolved, the document status changes to spread_complete.

API route

POST /api/spread/[id]/resolve-unmapped
Body: {

Section 07

Learning Loop: Storing, Retrieving & Attributing Manual Mappings

Design principle: The system must improve with every run. Every manual analyst decision is an investment in accuracy. The learning loop captures these decisions in a structured way that can be retrieved for future documents without requiring Claude, while always crediting the source of the learned knowledge explicitly in the output.

What is stored when an analyst confirms a mapping

Lookup algorithm — lib/spreading/learning-store.ts

// lib/spreading/learning-store.ts
export async function findLearnedMapping(
  canonicalField: string,
  rawLabel: string,
  templateType: string
): Promise<LearnedMapping | null> {
  // Priority 1: exact canonical_field + template_type match
  const exact = await db.query.learnedMappings.findFirst({
    where: and(
      eq(learnedMappings.canonical_field, canonicalField),
      eq(learnedMappings.template_type, templateType)
    ),
    orderBy: desc(learnedMappings.created_at),
  });
  if (exact) return exact;

  // Priority 2: canonical_field match on any template ('*')
  const anyTemplate = await db.query.learnedMappings.findFirst({
    where: and(
      eq(learnedMappings.canonical_field, canonicalField),
      eq(learnedMappings.template_type, '*')
    ),
    orderBy: desc(learnedMappings.created_at),
  });
  if (anyTemplate) return anyTemplate;

  // Priority 3: normalised raw label fuzzy match
  const normalisedLabel = normaliseLabelForLookup(rawLabel);
  const labelMatch = await db.query.learnedMappings.findFirst({
    where: ilike(learnedMappings.raw_label_pattern, \`%\${normalisedLabel}%\`),
    orderBy: desc(learnedMappings.created_at),
  });
  return labelMatch ?? null;
}

Section 10

Frontend: New Screens & Changes to Existing Screens

New screen A — Spread Review: /spread/[id]

Shows the full CoA-mapped spread output for a document after Stage 11 completes. This is the primary output the bank’s credit analyst uses.

  • Header: Document name + template + scope badge + balance check status (“✓ Balanced” in green, “✗ Imbalanced — $X difference” in red).
  • Statement selector tabs: Balance Sheet | P&L. Each shows all CoA entries for that statement in CoA ID order.
  • Spread table columns: CoA ID · Line item name · Raw label (source) · Year 1 value · Year 2 value · Confidence · Source badge (Claude / Learned / Manual) · Rationale (expandable tooltip).
  • Learning attribution: Rows with mapping_source = 'learned' show an amber “Learned” badge with a hover tooltip showing the full attribution message (“From Aditya Birla Finance 2024 · 2025-04-01”).
  • Unmapped banner: If unmapped_count > 0, a red banner at the top: “N items unmapped — spread output is incomplete. Resolve now →”
  • Override: Any row can be overridden by clicking its CoA ID and selecting a different entry from a searchable CoA dropdown. This triggers the override API and stores a new learned mapping.
  • Export: “Export Spread XLSX” primary button generates a multi-sheet workbook (BS sheet + P&L sheet + Confidence sheet + Learned Mappings sheet).

New screen B — Unmapped Resolver: /spread/[id]/unmapped

Dedicated screen for resolving unmapped items. Accessible from the Spread Review banner or the Document Library table (shown when status is has_unmapped).

  • Progress header: “Resolving N of M unmapped items” with a horizontal progress bar.
  • Item navigator: Left sidebar list of all unmapped items. Each shows raw label + statement type. Current item highlighted.
  • Item detail (centre panel): Raw label, canonical field, statement type, values, and Claude’s explanation of why it was not auto-mapped (“Ambiguity: 3 valid targets identified — [BS-009, BS-010, BS-083]”).
  • Suggestion cards (right panel): Up to 3 CoA candidate cards, each showing CoA ID + name, confidence score bar, CoA definition excerpt, Claude’s matching rationale. Clicking a card selects it (highlighted border). Analyst can also search the full CoA if none of the suggestions are right.
  • Rationale input: Pre-filled with Claude’s rationale for the selected suggestion. Analyst can edit before confirming. This text is stored in learned_mappings.rationale.
  • Action buttons: “Confirm mapping” (primary) · “Skip for now” (ghost) · “Next item” (after confirm).

COA-007Rows with confidence < 0.60 must be placed in unmapped_items and must NOT appear in the spread output until resolved by an analyst.MustDocument status set to 'has_unmapped'. Spread output is incomplete.

LRN-001Before calling Claude for COA mapping, the system must query learned_mappings for a matching entry. If found, apply it without a Claude call and increment times_applied.MustPriority: exact canonical_field + template > canonical_field + any_template > normalised label fuzzy match.
LRN-002When a learned mapping is applied, the coa_mappings.rationale must contain an explicit attribution message in the format specified in Section 07.MustAttribution must include: source company, year, date applied, original rationale, times previously applied.
LRN-003When an analyst overrides any mapping (whether Claude-generated or learned), the override must be stored as a new learned_mappings entry, and times_overridden must be incremented on the original learned mapping (if any).MustOverrides are how the system “unlearns” incorrect mappings over time.
LRN-004A learned mapping with times_overridden ≥ 2 must be automatically demoted: its learned_confidence must be set to 0.70 and it must no longer suppress the Claude call.ShouldPrevents a bad learning from repeatedly being applied incorrectly.
LRN-005A learned mapping that has been applied successfully ≥ 5 times with zero overrides must have its learned_confidence increased to 0.99.ShouldHigh-confidence established patterns deserve maximum confidence.
LRN-006The Spread Review screen must display a learning attribution tooltip on every row where mapping_source = 'learned'. The tooltip must include the source document, analyst, date, and original rationale.MustRequired for bank auditability. Analyst must be able to trace every learned decision.
SPR-001The Spread Review screen must present BS and P&L CoA entries in the exact order of their CoA IDs (BS-001 before BS-002, etc.), not in the order they appear in the source document.MustThis is the standardised order required by the bank’s credit system.
SPR-002The Spread XLSX export must contain: Sheet 1 = Balance Sheet (CoA order), Sheet 2 = P&L (CoA order), Sheet 3 = Confidence & Source (all rows with CoA ID, confidence, source, rationale), Sheet 4 = Learned Mappings Applied (attribution detail).MustSheet 4 is the audit trail for regulators and internal reviewers.
SPR-003Sign conventions from the CoA must be applied automatically during spreading. Contra-asset entries (e.g., BS-005 Bad Debt Reserve, BS-037 Accumulated Depreciation) must always be negative. Expense items must follow their stated sign convention.MustTrack in coa_reference.sign_convention and apply in the spread values calculation.
SPR-004If the same raw line item maps to a CoA entry that is already occupied by another row (e.g., two “revenue” items both mapping to PL-001), both values must be summed and a note added: “Aggregated from N source rows.”ShouldCommon when reports disaggregate revenue that the CoA aggregates.
